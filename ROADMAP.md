# ezvcc — Roadmap

ezvcc の開発を **5つのトラック** に分け、それぞれが独自のステージ(A, B, C, ...)で進んでいきます。トラック同士の進捗は同期しないため、機能を急ぐ時期と配布を整える時期を別々に動かせるようにしています。

設計・要件は [PLANNING.md](./PLANNING.md)、開発環境構築は [DEVELOPMENT.md](./DEVELOPMENT.md) を参照。

## 1. コア機能(音声処理)

> アプリ本来の役割 — マイクの声をリアルタイムに加工して出力する機能群

| ステージ | 内容 | 状態 |
|---------|------|------|
| **1-A** | 既定デバイスのパススルー(マイク → 既定の出力) + Start/Stop + 入力レベルメーター | ✅ 実装済み(Windows検証待ち) |
| **1-B** | ピッチ変更(-12 〜 +12 半音、SoundTouch導入)、バイパストグル | ⬜ |
| **1-C** | 入力デバイス選択UI(NAudioで列挙) | ⬜ |
| **1-D** | 出力デバイス選択UI、`CABLE Input (VB-Audio Virtual Cable)` 検出と推奨表示 | ⬜ |
| **1-E** | デバイス切断・例外発生時の自動復旧 / フォールバック | ⬜ |
| **1-F** | ノイズゲート、簡易EQ | 💭 |
| **1-G** | リバーブ、ロボットボイス等の追加エフェクト | 💭 |
| **1-H** | AI 声質変換(調査ベース) | 💭 |

**1-Bの完了基準**: スライダ操作で声の高さが滑らかに変化する。
**1-Dの完了基準**: 出力先を CABLE Input に設定 → Discord等で本アプリ加工後の声がマイク入力として聞こえる。

## 2. 配布・パッケージング

> ユーザーが「動かせる状態」になるまでの道のりを短くする

| ステージ | 内容 | 状態 |
|---------|------|------|
| **2-A** | 単一exe (self-contained) 配布の準備(`PublishSingleFile` 等の csproj 設定) | ✅ 設定済み |
| **2-B** | 初回リリース v0.1.0(Phase 1相当) を GitHub Releases に公開 | ⬜ |
| **2-C** | 起動時の SmartScreen 案内文を README に整備、必要なら署名なしの README 動画化 | ⬜ |
| **2-D** | Inno Setup で `Setup.exe` を並行配布(スタートメニュー登録 + アンインストーラ) | ⬜ |
| **2-E** | コード署名(EV または OV)で SmartScreen 警告を解消 | 💭 |
| **2-F** | 自動更新(Velopack / Squirrel.Windows のいずれか) | 💭 |
| **2-G** | MSIX パッケージ化 / Microsoft Store 配布 | 💭 |

**2-B の発火条件**: コア機能トラックが少なくとも 1-A まで Windows 実機で動作確認できていること。

## 3. ユーザビリティ・設定

> 「毎日使える」アプリにするための装備

| ステージ | 内容 | 状態 |
|---------|------|------|
| **3-A** | 設定の永続化(選択デバイス・ピッチ量・バイパスを `%AppData%\ezvcc\settings.json` に保存/復元) | ⬜ |
| **3-B** | プリセット(男声化 / 女声化 / 子供化 等)とプリセット切替UI | ⬜ |
| **3-C** | グローバルホットキー(Start/Stop/バイパス) | ⬜ |
| **3-D** | システムトレイ常駐、最小化時にトレイへ収納 | ⬜ |
| **3-E** | 起動時自動開始オプション(OSスタートアップ登録) | ⬜ |
| **3-F** | エラー時のユーザー向けダイアログ整備(技術的メッセージを和訳) | ⬜ |
| **3-G** | UIテーマ(ライト/ダーク追従)、多言語対応 | 💭 |

**3-A の発火条件**: 1-C / 1-D(デバイス選択UI)が動いていること(保存すべき値が出てくる)。

## 4. 音質・レイテンシ最適化

> 「聞こえる」だけでなく「気持ちよく使える」品質まで持っていく

| ステージ | 内容 | 状態 |
|---------|------|------|
| **4-A** | 既定構成(WASAPI Shared + デフォルトバッファ)での実測レイテンシ計測 → 目標 ~50ms 達成可否を判定 | ⬜ |
| **4-B** | バッファサイズ調整UI(ユーザーが latency vs 安定性をスライダ調整可) | ⬜ |
| **4-C** | キャプチャ/再生のフォーマット不一致時にResampler(NAudio標準)を挟む | ⬜ |
| **4-D** | WASAPI Exclusive モード切替オプション(低レイテンシ用) | ⬜ |
| **4-E** | パイプライン全体のジッタ計測ログ機能 | 💭 |
| **4-F** | ASIO 対応(プロ用途オーディオインターフェイス向け) | 💭 |
| **4-G** | マルチチャンネル対応(現状はモノラル/ステレオのみ) | 💭 |

**4-A は 1-A 完了後すぐ着手可能**。Windows 実機で計測する必要があるため、テスト機が用意できた時点で実施。

## 5. 品質・運用

> 持続可能に開発するための地盤

| ステージ | 内容 | 状態 |
|---------|------|------|
| **5-A** | ロジック層を `ezvcc.Core` プロジェクトに分離し、ユニットテストプロジェクト (`tests/ezvcc.Tests`) を立ち上げる | ⬜ |
| **5-B** | GitHub Actions Windows ランナーで `dotnet build` を CI 化(PR時/main push時) | ⬜ |
| **5-C** | リリースタグ push → `dotnet publish` → exe を Releases へ自動アップロード | ⬜ |
| **5-D** | クラッシュログの自動保存(`%LOCALAPPDATA%\ezvcc\logs\`) | ⬜ |
| **5-E** | 例外レポート画面(ユーザーが手動で送れる、テキストコピーできる) | 💭 |
| **5-F** | テレメトリ(オプトイン、Application Insights など) | 💭 |

**5-A の発火条件**: 1-B(ピッチ処理)着手前に分離しておくと、ピッチ周りのロジックが最初からテスト対象になる。MVP段階(1-A時点)では未分離でも問題なし。
